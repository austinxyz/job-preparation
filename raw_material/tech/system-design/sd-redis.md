# 系统设计复习：Redis 深度

> 覆盖主题：内存数据库本质 → 持久化（RDB / AOF）→ 数据结构选型（String / Bitmap / ZSet / List）→ 跳表 vs B+树 → 单线程模型与慢操作风险 → 高可用（Sentinel）→ 横向扩展（Redis Cluster / Hash Slot）→ 布隆过滤器（Bloom Filter / Counting Bloom Filter）

---

## 一、Redis 基础：为什么快，为什么不能替代一切

| # | 问题 | 核心答案 | 关键词 / 扩展点 |
|---|------|----------|----------------|
| 1 | Redis 为什么这么快？ | **纯内存操作**，数据存在 RAM，读写不需要磁盘 IO | 内存 vs 磁盘延迟差距约 1000 倍 |
| 2 | 为什么不用 Redis 替代所有数据库？ | ① 内存**易失性**：重启数据丢失；② **容量有限**：内存远比磁盘贵，无法存海量数据；③ 不支持复杂查询、多表 Join、强事务（ACID）| Redis = 高速缓存层，不是持久化数据库 |

---

## 二、持久化：RDB vs AOF

| 机制 | 全称 | 原理 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|------|----------|
| **RDB** | Redis Database | 定期对内存做**快照**，保存到磁盘 | 恢复速度快，文件紧凑 | 最后一次快照之后的数据可能**丢失** | 可容忍少量数据丢失（缓存场景） |
| **AOF** | Append Only File | 记录**每个写操作**追加到日志文件 | 数据完整，丢失少 | 文件大，恢复速度慢 | 不允许数据丢失的场景 |
| **RDB + AOF 混合** | — | RDB 快速恢复大部分数据，AOF 补上增量 | 兼顾恢复速度和数据完整性 | 配置稍复杂 | **生产环境推荐配置** |

## 三、数据结构选型

| 数据结构 | 特点 | 典型场景 |
|----------|------|----------|
| **String** | 存字符串或数字；`INCR` 原子加减 | 视频播放量、点赞数、分布式锁 |
| **Bitmap** | 每个 bit 对应一个用户/状态；10 亿用户只需约 125MB | **用户在线状态**（在线=1，离线=0） |
| **ZSet（Sorted Set）** | 每个 member 有 score，自动按 score 排序 | **实时排行榜**（按播放量排名） |
| **List** | 有序双端队列；`LPUSH` 头插，`LTRIM` 保留最近 N 个 | **用户 Feed 缓存**（最近 50 条推荐，按时间顺序） |
| **Set** | 无序集合，去重 | 标签、关注列表（不需要顺序） |
| **HyperLogLog** | 概率算法，估算集合基数（UV）；有误差 | 估算视频**独立观看人数**（不是 membership 查询）|

### 四个场景速查

| 场景 | 数据结构 | 理由 |
|------|----------|------|
| 视频实时播放量 | **String（INCR）** | 原子加 1，简单高效 |
| 用户是否在线（10 亿用户）| **Bitmap** | 每用户 1 bit，10 亿用户仅 125MB |
| 热门视频排行榜（实时更新）| **ZSet** | score = 播放量，`ZREVRANGE` 取 Top K |
| 用户 Feed 缓存（最近 50 个）| **List** | 有序、LPUSH + LTRIM 维护固定长度 |

> **常见误区**：Feed 缓存不要用 Set（无序）；排行榜不要用 HyperLogLog（只能估算基数，不能精确排序）。

## 四、ZSet 底层：跳表 vs B+树

| # | 问题 | 核心答案 |
|---|------|----------|
| 3 | ZSet 底层用什么数据结构实现？ | **跳表（Skip List）**：多层有序链表，查找/插入/删除都是 O(log n) |
| 4 | 为什么用跳表而不用 B+树？ | **跳表**适合纯内存操作：节点通过指针连接，插入删除只需改指针，无需页分裂，实现简单；**B+树**为磁盘 IO 优化，节点对应磁盘页，减少 IO 次数。Redis 是内存数据库，无需考虑磁盘 IO，跳表更轻量合适 |

---

## 五、单线程模型与慢操作风险

| # | 问题 | 核心答案 | 关键词 |
|---|------|----------|--------|
| 5 | Redis 单线程为什么还能这么快？ | 所有操作都在内存里，本身极快；单线程**避免了锁竞争和线程上下文切换**，这两项在多线程系统里开销很大 | 无锁 = 零竞争开销 |
| 6 | 单线程有什么风险？ | **一个慢操作会阻塞所有其他请求**。如在生产环境执行 `KEYS *`（扫描全部 key），Redis 会卡死数秒，所有依赖 Redis 的服务全部超时 | `KEYS *` 是生产事故元凶 |
| 7 | 需要扫描所有 key，安全的替代方案是什么？ | 用 **`SCAN` 命令**：每次只扫描一小批，通过 cursor 分批迭代，每次调用都很快，不会长时间阻塞 | `SCAN` = 渐进式扫描，生产环境必用 |

## 六、高可用：Redis Sentinel

| # | 问题 | 核心答案 |
|---|------|----------|
| 8 | Redis 主节点挂了怎么办？ | **Redis Sentinel（哨兵）**：多个 Sentinel 节点持续向 Redis 主节点发 ping；若 **Quorum**（多数哨兵）都认为主节点不可用，触发 Leader Election，选举新的从节点升为主节点，并通知所有客户端更新连接 |
| 9 | 为什么要用 Quorum 投票而不是单个 Sentinel 判断？ | 防止单个 Sentinel 网络抖动导致**误判**（false positive），触发不必要的故障切换 —— 与 Kafka ISR 的多数派确认思想相同 |

> **Sentinel 解决的问题**：高可用（HA）——主节点挂了自动切换，不解决数据量问题。

---

## 七、横向扩展：Redis Cluster 与 Hash Slot

| # | 问题 | 核心答案 |
|---|------|----------|
| 10 | 单台 Redis 内存装不下（如需存 1TB 数据），Sentinel 够用吗？ | 不够。Sentinel 只解决 HA，不解决容量。需要 **Redis Cluster**：把数据分散到多个节点，每个节点负责一部分数据 |
| 11 | Redis Cluster 用什么分片算法？和一致性 Hash 有什么区别？ | Redis Cluster 用 **Hash Slot**：将所有 key 映射到 16384 个 slot，再将 slot 分配给节点。**优点**：简单可预测，节点增减只需迁移对应 slot，运维方便；**缺点**：slot 总数固定（16384），扩展性不如一致性 Hash，但 Redis 场景通常不需要频繁大规模扩缩容 |

### Sentinel vs Redis Cluster 对比

| | Redis Sentinel | Redis Cluster |
|---|---|---|
| **解决问题** | 高可用（HA）| 数据量 + 横向扩展 |
| **机制** | 主从复制 + 自动故障切换 | 数据分片（Hash Slot）|
| **能否同时使用** | ✅ 互补，不互斥 |
| **生产标准配置** | Cluster 多节点，每个节点有 Sentinel 监控的主从复制 |

## 八、布隆过滤器（Bloom Filter）

**场景**：10 亿用户 × 100 亿视频，判断"用户是否看过某个视频"

- 用 Bitmap 每用户存 100 亿 bit ≈ 1.25GB，10 亿用户 = **1.25EB，完全不可行**
- 用数据库查询，规模太大，延迟无法接受

### 布隆过滤器原理

```
添加元素：对元素做 k 次哈希，将对应 k 个 bit 置为 1
查询元素：对元素做 k 次哈希，检查对应 k 个 bit 是否全为 1
          → 全为 1：可能存在（有假阳性）
          → 有 0：一定不存在（无假阴性）
```

| 特性 | 说明 |
|------|------|
| **假阳性（False Positive）** | hash 碰撞导致误判"存在"——可接受（顶多少推一个视频）|
| **无假阴性（No False Negative）** | bit 为 0 就绝对不存在 |
| **不支持删除** | 多个元素共享同一 bit，删除会影响其他元素 |
| **不支持精确查询** | 只能回答是/否，不能列举元素 |

### 改进：Counting Bloom Filter

| 需求 | 方案 |
|------|------|
| 需要支持删除 | **Counting Bloom Filter**：每个 bit 位改为计数器，添加时 +1，删除时 -1，计数为 0 才判为不存在 |
| 代价 | 每个位置存整数而非 bit，内存增加，但支持了删除操作 |

## 九、术语速查

| 术语 | 中文 | 一句话解释 |
|------|------|-----------|
| RDB | 快照持久化 | 定期把内存数据存成二进制快照文件 |
| AOF | 追加日志持久化 | 每个写操作追加到日志，重启时回放恢复 |
| Skip List | 跳表 | 多层有序链表，O(log n) 查找，ZSet 底层实现 |
| Sentinel | 哨兵 | 监控 Redis 主从，Quorum 投票后自动故障切换 |
| Redis Cluster | 集群 | 官方分布式方案，Hash Slot 分片，解决容量问题 |
| Hash Slot | 哈希槽 | Redis Cluster 的分片单位，共 16384 个 |
| SCAN | 渐进式扫描 | `KEYS *` 的安全替代，分批迭代不阻塞 |
| Bloom Filter | 布隆过滤器 | 概率型数据结构，省内存判断 membership，有假阳性，无假阴性 |
| Counting Bloom Filter | 计数布隆过滤器 | Bloom Filter 改进版，支持删除操作 |
| HyperLogLog | — | 估算集合基数（UV），不支持 membership 查询 |

## 十、面试答题习惯提醒

> AI 反馈：本节两个易错点：

1. **Feed 缓存用 List 不用 Set**：Feed 有时间顺序，Set 无序。选数据结构前先想清楚"需不需要顺序"。

2. **大规模 Bitmap 要做内存估算**：10 亿用户 × 100 亿视频的 Bitmap 场景，要立刻心算出 1.25EB 不可行，主动说"这规模下 Bitmap 不合适，应该用 Bloom Filter"，体现数字敏感度。
