# 系统设计复习：Sharding 与扩展性

> 覆盖主题：垂直/水平扩展 → 取模Hash → 一致性Hash → 虚拟节点 → Hot Key → 读写分离 → Replica延迟 → 容量估算 → 综合设计

---

## 一、扩展策略

| # | 问题 | 核心答案 | 关键词 / 扩展点 |
|---|------|----------|----------------|
| 1 | 垂直扩展 vs 水平扩展，如何选？ | **垂直**：换更强服务器，简单无需改代码，但硬件有上限，不可持续；**水平**：加更多服务器，扩展性强，但需要设计分区策略，迁移风险高 | Scale Up vs Scale Out；初期用垂直，到达瓶颈再水平 |
| 2 | 水平扩展数据库时，如何决定数据放哪台服务器？ | 选择 **Shard Key**（分片键），对 shard key 做 hash 后路由到对应服务器。shard key 要根据查询模式选（按 user_id 还是 username 取决于业务主要查询方式） | Shard Key 选择是最关键决策，选错后期迁移代价极高 |

---

## 二、Hash 策略

| # | 问题 | 核心答案 | 关键词 / 扩展点 |
|---|------|----------|----------------|
| 3 | 取模 Hash（`user_id % N`）的缺点？ | 加减服务器时，N 改变导致**几乎所有数据的路由结果改变**，需要大规模数据迁移 | 10台→11台：约90%的数据需要迁移 |
| 4 | 一致性 Hash 如何解决取模 Hash 的问题？ | 将所有服务器映射到一个 **Hash 环**上，数据 hash 后找顺时针最近的服务器。加减节点时只影响**相邻节点**的数据，迁移量大幅减少 | Hash Ring；加一台新节点只影响其前驱节点的数据 |
| 5 | 一致性 Hash 如何处理服务器能力不均？ | 引入**虚拟节点（Virtual Nodes）**：能力强的服务器在环上放更多虚拟节点（如普通1个，强机器3个），承担更多数据 | VNode；权重分配；Cassandra、DynamoDB 都用虚拟节点 |
| 6 | 一致性 Hash 解决不了的问题是什么？ | **Hot Key（热点键）**：某个数据（如明星账号）被访问频率极高，无论 Hash 多均匀，该数据永远落在同一台服务器上，造成单点过热 | Hash 解决的是**均匀分布**，不解决**访问频率不均** |

## 三、Hot Key 处理

| # | 问题 | 核心答案 | 关键词 / 扩展点 |
|---|------|----------|----------------|
| 7 | Hot Key 的三种处理方案？ | ① **加缓存**（Redis）：热点数据放内存，减少DB压力；② **多读 Replica**：增加读节点分摊读流量；③ **读写分离**：写走主节点，读走 Replica | 三者通常组合使用；缓存是第一道防线 |
| 8 | 三种方案的局限性分别是什么？ | **缓存**：失效瞬间 Cache Stampede；**多读 Replica**：主从复制延迟，可能读到旧数据；**读写分离**：复制延迟问题同上，且强一致场景不适用 | 没有银弹，需要根据业务一致性要求组合使用 |
| 9 | 明星更新头像后，缓存+读写分离场景下，完整读请求路径和哪些节点可能读到旧数据？ | 读请求路径：① **读缓存**（可能旧头像，更新后需删除缓存）→ ② **缓存 miss，读 Replica**（复制延迟，可能旧头像）→ ③ **Replica 同步完成后读正确**，更新缓存。写节点更新后：先删缓存（防 stale 写回），但 Replica 延迟窗口内仍有旧数据风险 | 旧数据出现的两个节点：缓存（TTL 或删除前）、Replica（复制延迟期间） |
| 10 | 缓存 + Replica 延迟的双重竞态：删缓存后，有人从 Replica 读到旧数据写回缓存怎么办？ | ① **Cache miss 强制读主节点**：一致性高，但主节点压力大；② **设短 TTL**：允许短暂旧数据窗口，TTL 到期自动修正 | 一致性和性能再次取舍：关键业务用方案①，资讯类用方案② |

## 四、容量估算（Back-of-envelope Estimation）

| # | 问题 | 核心答案 | 关键词 / 扩展点 |
|---|------|----------|----------------|
| 11 | 面试拿到需求，第一步问什么？ | ① **系统规模**（DAU、数据量）；② **非功能需求**（高可用、低延迟、强/最终一致性）。规模决定方案，非功能需求决定设计取舍 | 先澄清需求再出方案；面试官给出 DAU 后立刻推算 TPS |
| 12 | 1亿 DAU 如何粗略估算 TPS？ | 1亿 / 86400秒 ≈ **1000 TPS（平均值）**。峰值 = 平均值 × 3~10 倍 ≈ **3000~10000 TPS**。读写比100:1时：峰值读 ≈ 9900 TPS，峰值写 ≈ 100 TPS | 86400 ≈ 10万秒（面试口算用）；峰值系数 3~10x |
| 13 | 1亿 DAU、读写比100:1，推导出哪些架构决策？ | **写少读多** → 读写分离 + 多读 Replica；读加**缓存**（Redis/CDN）；服务层需要 **API Gateway + 负载均衡**；峰值万级TPS需要**水平扩展** | TPS → 服务器数量：单机 QPS 约 1~5万，可估算需要几台 |

## 五、综合：微博明星发动态全链路

| 阶段 | 组件 | 关键问题 | 解决方案 |
|------|------|----------|----------|
| 写入 | 客户端 → API Gateway → 负载均衡 → 后台服务 | 高峰请求量大 | 负载均衡 + 水平扩展服务层 |
| 存储 | 后台服务 → 按 user_id Shard → 写主数据库 | 哪台 DB？ | 一致性 Hash 路由 |
| 缓存失效 | 写成功后删除缓存 | 竞态条件 | Double Delete + 短 TTL 兜底 |
| 主从同步 | 主DB → 读 Replica 异步同步 | 复制延迟期间读到旧数据 | 接受短暂不一致（资讯类 AP） |
| 粉丝分发 | 5000万粉丝收到动态 | 推5000万次 or 每次拉？ | **混合模式**：普通用户推（Kafka异步），名人用拉 |
| 读扩展 | 粉丝刷 Feed | Hot Key：明星数据集中一台 | 多读 Replica + Redis 缓存 |

**关键决策一句话**：微博是 AP 系统，读写比高，用读写分离+缓存抗读流量，用 Kafka 异步推送抗写扩散，名人动态用拉模式避免写风暴。

## 六、术语速查

| 术语 | 中文 | 一句话解释 |
|------|------|-----------|
| Sharding | 分片 | 将数据水平拆分到多台服务器 |
| Shard Key | 分片键 | 决定数据路由到哪台服务器的字段 |
| Consistent Hashing | 一致性Hash | Hash环结构，加减节点只影响相邻数据 |
| Virtual Node | 虚拟节点 | 一台服务器在Hash环上占多个位置，解决能力不均 |
| Hot Key | 热点键 | 单条数据访问量极高，导致所在服务器过载 |
| Read Replica | 读副本 | 主节点的只读副本，分担读流量 |
| Replication Lag | 复制延迟 | 主从同步的时间差，期间读可能得到旧数据 |
| Scale Up | 垂直扩展 | 换更强的单台服务器 |
| Scale Out | 水平扩展 | 增加更多服务器 |
| DAU | 日活用户数 | Daily Active Users，容量估算起点 |
| TPS | 每秒事务数 | Transactions Per Second，衡量系统吞吐量 |
| Back-of-envelope | 粗略估算 | 面试中快速推算数量级，不需要精确 |

## 七、面试答题习惯提醒

> AI 反馈：目前 mid-level 水平，距离 senior 差两个习惯：

1. **主动说 trade-off**：给出方案后立刻说"这个方案的缺点是……，但在本场景下可以接受，因为……"
2. **主动给完整方案**：不等面试官追问。如推拉模式，直接说"普通用户推，名人拉，混合模式"，不要等被提醒
