# 系统设计复习：网络基础

> 覆盖主题：DNS 层级解析 → TCP 三次握手（SYN Flood）→ HTTPS/TLS（CA/OCSP）→ 实时通信（轮询/SSE/WebSocket）→ CDN（GeoDNS/动态加速）→ 级联故障（Circuit Breaker/Thundering Herd）

---

## 一、DNS 解析

| # | 问题 | 核心答案 | 关键词 / 扩展点 |
|---|------|----------|----------------|
| 1 | 浏览器访问 `www.google.com`，DNS 解析经过哪些步骤？ | ① 浏览器本地缓存 → ② OS 缓存 → ③ 本地 DNS（ISP）→ ④ Root NS（返回 .com 地址）→ ⑤ TLD NS（返回 google.com 地址）→ ⑥ Authoritative NS（返回最终 IP）| 层级查询；实际请求通常在第②③步命中缓存即返回 |
| 2 | DNS 缓存的 TTL 是什么？迁移服务器时为什么要提前降低 TTL？ | TTL（Time To Live）= 缓存有效期。迁移前提前将 TTL 从 24 小时降到 5 分钟，迁移完成后新 IP 最多 5 分钟内全网生效，避免用户长时间访问旧 IP | TTL 越小，更新越快，但 DNS 查询流量增加 |

## 二、TCP 三次握手与 SYN Flood

| # | 问题 | 核心答案 | 关键词 / 扩展点 |
|---|------|----------|----------------|
| 3 | TCP 为什么是三次握手，两次不够吗？ | 两次握手存在**半开连接（Half-Open）**风险：服务器收到 SYN 后立即分配资源，但客户端从未收到 SYN-ACK（网络延迟/丢包），服务器资源白白占用。三次握手确保双方都能收发，才正式建立连接 | 三次握手 = 双向确认；防止服务器单方面分配资源 |
| 4 | SYN Flood 攻击的原理是什么？ | 攻击者发送海量 SYN 包，但故意不发第三次 ACK，服务器维护大量半开连接队列直至超时，正常用户的连接请求被挤满无法处理 | DDoS 的一种；攻击点：服务器的半开连接队列有限 |
| 5 | 如何防御 SYN Flood？ | ① **SYN Cookie**：服务器不立即分配连接状态，而是用加密算法生成 Cookie 放在 SYN-ACK 中，只有收到合法 ACK 才建立连接，伪造源 IP 的包无法完成握手；② **高防节点（Anti-DDoS）**：在边缘清洗恶意流量，只将合法连接转发给业务服务器 | SYN Cookie = 零资源消耗直到握手完成；高防节点 = 流量清洗 |

## 三、HTTPS / TLS

| # | 问题 | 核心答案 | 关键词 / 扩展点 |
|---|------|----------|----------------|
| 6 | HTTPS 的握手过程是什么？为什么既用非对称加密又用对称加密？ | ① 客户端发 Client Hello → ② 服务器返回证书（含公钥）→ ③ 客户端验证 CA → ④ 客户端用公钥加密生成**会话密钥（Session Key）**→ ⑤ 双方用会话密钥对称加密通信。非对称加密安全但慢，只用于交换密钥；对称加密快，用于实际数据传输 | 非对称加密 = 安全交换密钥；对称加密 = 高效传输数据 |
| 7 | 浏览器怎么验证服务器证书是合法的，而不是中间人伪造的？ | 证书由**CA（Certificate Authority）**签发，CA 用自己的私钥对证书签名。浏览器内置受信任 CA 的公钥，验证签名即可确认证书合法。中间人没有 CA 私钥，无法伪造有效签名 | CA 信任链（Chain of Trust）；浏览器内置根证书 |
| 8 | 证书如果被吊销（比如私钥泄露），如何让全网快速知道？什么是 OCSP Stapling？ | **OCSP（Online Certificate Status Protocol）**：客户端实时向 CA 查询证书状态，但每次建立连接都要额外请求 CA，增加延迟。**OCSP Stapling**：服务器提前向 CA 获取带时间戳的 OCSP 响应，在 TLS 握手时一起发给客户端，省去客户端额外查询，速度更快 | OCSP Stapling = 服务器代替客户端做吊销查询，TLS 握手时附带结果 |

## 四、实时通信：轮询 / SSE / WebSocket

| # | 问题 | 核心答案 | 关键词 / 扩展点 |
|---|------|----------|----------------|
| 9 | HTTP 轮询的缺点是什么？ | 客户端定期发请求，大多数时候服务器没有新数据，**无效请求多、服务器压力大、实时性差**（取决于轮询间隔） | 简单易实现；适合实时性要求低、连接数少的场景 |
| 10 | SSE（Server-Sent Events）是什么？适合什么场景？ | 客户端发一次 HTTP 请求，服务器**保持连接不断开**，持续推送数据给客户端。**单向**（服务器→客户端）。适合股票行情、通知推送等**服务器单向推送**场景 | SSE = 持久 HTTP 连接；单向 = 客户端发消息还需单独 HTTP 请求 |
| 11 | WebSocket 和 SSE 的核心区别是什么？WebSocket 如何建立连接？ | WebSocket 是**全双工双向通信**。建立方式：客户端发带 `Upgrade: websocket` 头的 HTTP 请求 → 服务器返回 `101 Switching Protocols` → 连接升级为 WebSocket | WebSocket Upgrade；适合聊天、多人游戏等双向实时场景 |
| 12 | 微信 1 亿同时在线用户，每人一个 WebSocket 长连接，有什么挑战？ | ① **资源消耗**：现代服务器用**异步非阻塞 IO**（Node.js/Netty），一个线程管理数万连接，而非一连接一线程；② **跨服务器消息路由**：用户 A 和用户 B 在不同 WebSocket 服务器上，用 **Redis pub/sub** 广播（轻量快速）或 **Zookeeper** 服务注册（强一致性）转发消息 | 异步非阻塞 IO = 解决线程瓶颈；Redis pub/sub = 跨服务器消息路由 |
| 13 | 用户断线重连到新服务器，消息会丢吗？如何保证幂等？ | 断线后新服务器重新订阅该用户的消息频道；旧服务器收到的消息未 ACK 则重发（At-least-once）；**给每条消息唯一 message_id**，客户端收到相同 ID 的消息直接丢弃，保证幂等 | 与消息队列的 At-least-once + 幂等思路完全相同 |

## 五、CDN

| # | 问题 | 核心答案 | 关键词 / 扩展点 |
|---|------|----------|----------------|
| 14 | CDN 的核心机制是什么？纽约用户访问北京服务器的图片，请求路径如何？ | ① 纽约用户请求图片 → ② DNS 返回纽约 CDN 节点 IP（**GeoDNS**，按地理位置返回最近节点）→ ③ CDN 有缓存则直接返回；没有则**回源**到北京拿，缓存后返回 | 就近缓存，回源按需；CDN 命中率越高，延迟越低 |
| 15 | CDN 对动态内容（聊天消息、实时价格）有用吗？ | 动态内容变化快，**缓存意义不大**。但 CDN 对动态内容仍有价值：用户到 CDN 节点的 TCP 连接延迟低，CDN 到源服务器之间走**优化专线**，整体延迟仍可降低（**协议加速 / TCP 连接优化**） | Cloudflare 等 CDN 对动态请求也有加速；CDN ≠ 只是缓存 |

## 六、级联故障：Circuit Breaker 与 Thundering Herd

| # | 问题 | 核心答案 | 关键词 / 扩展点 |
|---|------|----------|----------------|
| 16 | 数据库变慢，为什么会导致整个系统雪崩？ | 数据库慢 → 订单服务线程全部**同步阻塞**等待 → 订单服务线程耗尽，新请求堆积 → API 服务调用订单服务超时，线程也开始阻塞 → 整个调用链雪崩。根源：**同步阻塞调用让一个慢节点拖垮整条链路** | 级联故障（Cascading Failures）= 慢节点通过同步调用蔓延 |
| 17 | Circuit Breaker（熔断器）如何阻止级联故障？三个状态如何转换？ | **Closed（正常）**：正常放行请求；失败率超阈值 → **Open（熔断）**：直接快速失败，不再调用下游，保护上游；经过一段时间 → **Half-Open（试探）**：放少量请求探测是否恢复；试探成功 → 回 Closed；试探失败 → 回 Open | Fail Fast（快速失败）= Circuit Breaker 核心思想；自动检测和恢复，无需人工介入 |
| 18 | 数据库恢复后，积压的百万请求同时涌入会怎样（Thundering Herd）？Circuit Breaker 如何应对？ | 百万请求瞬间冲击 → 刚恢复的数据库再次崩溃。**Half-Open 状态只放少量请求**，让数据库和缓存慢慢预热，逐步扩大流量，避免二次雪崩 | Thundering Herd = Cache Stampede 的系统级版本；Half-Open 限流是关键防护 |

## 七、三种实时通信方案对比

| 方案 | 方向 | 连接方式 | 适用场景 | 不适用场景 |
|------|------|----------|----------|------------|
| **HTTP 轮询** | 客户端主动拉 | 短连接（每次新建） | 实时性低、客户端少、实现简单 | 高频更新、大量并发 |
| **SSE** | 服务器→客户端（单向） | 持久 HTTP 长连接 | 股票行情、通知推送、新闻feed | 需要客户端向服务器发消息 |
| **WebSocket** | 双向 | HTTP Upgrade → 全双工 | 聊天、多人游戏、协作编辑 | 只需单向推送（SSE 更简单） |

## 八、术语速查

| 术语 | 中文 | 一句话解释 |
|------|------|-----------|
| TTL | 生存时间 | DNS 缓存有效期，过期后重新查询 |
| SYN Flood | SYN 洪泛攻击 | 发海量 SYN 耗尽服务器半开连接队列 |
| SYN Cookie | SYN 饼干 | 不提前分配资源，用加密 Cookie 验证握手合法性 |
| CA | 证书颁发机构 | 用私钥签发 TLS 证书，建立信任链 |
| OCSP Stapling | OCSP 装订 | 服务器预取证书吊销状态，TLS 握手时附带，减少客户端额外查询 |
| Session Key | 会话密钥 | TLS 握手后生成的对称密钥，用于加密实际通信 |
| SSE | 服务器推送事件 | 基于持久 HTTP 连接的单向实时推送 |
| WebSocket Upgrade | WebSocket 升级 | HTTP 101 协议升级，建立全双工连接 |
| GeoDNS | 地理 DNS | 根据用户地理位置返回最近 CDN 节点 IP |
| Cache Miss / Origin Pull | 缓存未命中 / 回源 | CDN 没有缓存时，去源服务器拉取资源 |
| Circuit Breaker | 熔断器 | 失败率超阈值时快速失败，保护上游服务 |
| Thundering Herd | 雷群问题 | 服务恢复后大量积压请求同时冲击，可能再次压垮 |
| Cascading Failures | 级联故障 | 慢节点通过同步阻塞调用蔓延，整个系统雪崩 |

## 九、面试答题习惯提醒

> AI 反馈：本节两个需主动说出的扩展点：

1. **CDN 对动态内容也有价值**：不要说"动态内容不能用 CDN"就停住。要主动补充：CDN 虽然不缓存动态内容，但通过用户→CDN 低延迟 TCP 连接 + CDN→源站优化专线，仍能降低整体延迟。这是面试中体现深度的机会。

2. **WebSocket 跨服务器路由要主动说出两种方案**：Redis pub/sub（轻量，适合大多数场景）vs Zookeeper（强一致性，适合服务发现），并说明各自适用条件，不要等面试官追问。
