# 系统设计复习：设计抖音（短视频平台）

> 覆盖主题：需求澄清 → 数据模型 → API 设计（Presigned URL）→ 上传 + 转码流水线 → Feed 播放链路 → 缓存策略（Hot Key / Cache Stampede）→ 数据库分片 + KV 反范式 → 流媒体优化（HLS / ABR / Prefetch）→ 推荐系统（召回 + 排序 / 完播率 / Exploration vs Exploitation）

---

## 一、需求澄清（面试第一步）

**功能性需求（核心）**
- 上传视频（最大 500MB）
- 刷 Feed（无限滚动，暂时按时间顺序）
- 观看视频（要求 2 秒内开始播放）
- 点赞评论（次优先级）

**非功能性需求**
| 指标 | 要求 |
|------|------|
| DAU | **5 亿** |
| 视频起播延迟 | **< 2 秒** |
| 可用性 | **99.99%** |
| 读写比 | **~1000:1**（读远多于写） |

> **面试金句**：抖音是极度读多写少的系统，架构重点在于如何用缓存 + CDN 扛读压力，写入链路用异步消息解耦。

---

## 二、核心实体与数据模型

| 实体 | 存储位置 | 原因 |
|------|----------|------|
| **Video Metadata** | 关系型数据库（分库分表） | 结构化字段，支持查询 |
| **Video File** | 对象存储（S3）| 二进制大文件，不适合存数据库 |
| **User** | 关系型数据库 | 结构化，需要关联查询 |
| **User → Video 列表** | KV 存储（如 Redis / DynamoDB）| 反范式，O(1) 查询用户所有视频 |

**Video Metadata 关键字段**：`video_id`, `user_id`, `title`, `category`, `location`, `created_at`, `status`（processing/ready）, `url_360p`, `url_720p`, `url_1080p`

---

## 三、API 设计

| API | 说明 | 关键设计 |
|-----|------|----------|
| `POST /api/upload` | 上传请求，body 含 metadata | 返回 **Presigned URL**，不直接上传到业务服务器 |
| `PUT /{presignedURL}` | 客户端直传视频文件到 S3 | 绕过业务服务器，节省带宽和服务器压力 |
| `GET /api/feeds?cursor=&limit=` | 获取 Feed 列表 | 返回每条视频的 metadata + **直接可用的 Streaming URL**（CDN URL），客户端无需再请求下载链接即可预加载 |

> **关键设计**：Feed API 返回的每个条目必须包含可直接播放的 streaming URL，客户端拿到响应即可开始预加载下一批视频，不需要额外请求。

---

## 四、上传与转码流水线

| 步骤 | 组件 | 说明 |
|------|------|------|
| 1 | 客户端 → API Gateway → 业务服务 | 保存 metadata（status = processing），返回 Presigned URL |
| 2 | 客户端 → S3 | 直接上传原始文件，不经过业务服务器 |
| 3 | S3 → 消息队列 | S3 上传完成事件触发消息 |
| 4 | 转码服务消费消息 | 视频切成 chunk，**并行转码**多种格式（360p/720p/1080p）和 HLS 格式 |
| 5 | 转码完成 → S3 | 转码后文件存回 S3 |
| 6 | 消息队列 → Metadata 更新服务 | 更新 status = ready，写入各画质 URL |
| 7 | 消息队列 → KV 更新服务 | 将新 video_id 追加到 user 的视频列表 |

**关键问答**

| # | 问题 | 核心答案 |
|---|------|----------|
| 1 | 转码耗时几分钟，用户能马上看到自己的视频吗？ | 不能，视频 status = processing 期间对其他用户不可见；**分块并行转码**可缩短等待时间（把一个大文件切成几十个 chunk，多个 worker 同时处理） |
| 2 | 为什么用消息队列而不是同步调用触发转码？ | 转码耗时长（分钟级），同步等待会阻塞上传接口；消息队列解耦，失败可重试，天然幂等 |

## 五、Feed 播放链路

```
用户打开抖音
    → API Gateway / Load Balancer
    → Feed 服务（查推荐系统）
    → Redis 缓存（热门视频 metadata）
    → 数据库（缓存未命中时回源）
    → 返回视频列表（含 CDN Streaming URL）
    → 客户端预加载视频 chunk（从最近 CDN 节点）
```

| # | 问题 | 核心答案 | 关键词 |
|---|------|----------|--------|
| 3 | 5 亿 DAU 不断刷 Feed，数据库会被压垮，怎么办？ | 加 **Redis 缓存**：缓存热门视频 metadata；每个用户的个性化 Feed 变化快且各不相同，不适合缓存 | 缓存热门 metadata；个性化 Feed 不缓存 |
| 4 | 用户在纽约，视频存在北京 S3，怎么解决延迟？ | **CDN**：视频文件缓存到距用户最近的边缘节点（GeoDNS 路由） | CDN = 视频分发的核心基础设施 |
| 5 | 某视频病毒式传播，Redis 里的 metadata 缓存突然失效，几千万请求冲向数据库？ | **Cache Stampede**：用 **Request Coalescing（单飞/Mutex）**，只允许一个请求去数据库重建缓存，其他请求等待结果；也可用 Circuit Breaker 保护 | Cache Stampede = 缓存击穿；单飞锁比 Circuit Breaker 更轻量直接 |
| 6 | 病毒视频造成 Redis Hot Key，该分片压力过大？ | 增加该 key 的 **replica**，将读流量分散到多个副本 | Hot Key → 读副本分散 |

## 六、数据库设计

| # | 问题 | 核心答案 | 关键词 |
|---|------|----------|--------|
| 7 | 100 亿视频的 metadata 表，用什么做分片键？ | **video_id 哈希分片**：负载最均匀。用 user_id 分片会导致热门博主所在分片成热点；用 created_at 分片会导致当前时间段分片成热点 | 哈希分片 = 均匀分布；避免热点是首要原则 |
| 8 | 用户打开个人主页，要查自己所有视频——video_id 分片后视频分散在所有分片，怎么办？ | 维护独立的 **KV 存储**：`user_id → [video_id 列表]`，查询 O(1)，不需要扫全分片。写多付一次代价，换查询高效 | 反范式（Denormalization）= 用空间换时间 |
| 9 | 上传视频后，metadata 写入数据库 + KV 更新用户视频列表，两步如何保证一致性？ | 业务上可以接受**最终一致性**（视频上传后几秒才出现在个人主页，用户可接受）；用**消息队列**异步更新 KV，比定时同步更实时，且天然支持重试和幂等 | 最终一致性 > 强事务（业务允许时）；消息队列 > 定时同步（实时性更好） |

## 七、流媒体优化（视频秒开）

| 技术 | 说明 |
|------|------|
| **HLS 分片** | 视频切成小 chunk（通常 2-10 秒），边下边播，不需等整个文件下载完 |
| **CDN 就近缓存** | chunk 缓存到离用户最近的节点，降低延迟 |
| **自适应码率（ABR）** | 客户端实时检测网速，动态切换 360p/720p/1080p，网络差时降码率保流畅 |
| **预加载（Prefetching）** | 用户在看第 N 个视频时，后台已开始下载第 N+1、N+2 个的前几个 chunk；甚至在用户打开 App 前，根据行为规律提前将可能看的视频预加载到 CDN |
| **Feed 预加载时机** | 用户刷到列表过半，立刻请求下一批 Feed（含 streaming URL），保证无限滚动不卡顿 |

> **权衡**：Prefetching 会浪费带宽（预测错了的视频白下载）。对于行为规律强（固定时间看固定类型）的用户值得预取；行为随机的用户效益低。

## 八、推荐系统

### 信号类型

| 类型 | 例子 | 准确性 |
|------|------|--------|
| **显式信号** | 点赞、评论、分享、关注 | 明确，但用户可能随手操作 |
| **隐式信号** | 观看时长、**完播率**、是否跳过、刷新频率 | **更准确**——是用户的下意识行为，骗不了人 |

> **核心指标**：抖音推荐系统最重要的指标是**完播率**，而不是点赞数。看了 3 遍不点赞 > 随手点赞看了 3 秒。

### 推荐两阶段

| 阶段 | 目标 | 方法 | 规模 |
|------|------|------|------|
| **召回（Recall）** | 从 100 亿视频快速筛出几千个候选 | 简单规则过滤：时间、地域、历史喜好类别 | 快速，O(规则数) |
| **排序（Ranking）** | 对几千个候选精细打分，取 Top K | 复杂 ML 模型，结合显式 + 隐式信号 | 精准，计算密集 |

> **为什么分两阶段**：100 亿视频全部用复杂模型打分，实时计算不可能完成；召回先粗筛，排序再精选，兼顾速度和质量。

### Exploration vs Exploitation（探索 vs 利用）

| 策略 | 做法 | 效果 |
|------|------|------|
| **Exploitation（利用）** | 只推用户已知喜欢的内容 | 短期点击率高，但用户落入**信息茧房（Filter Bubble）** |
| **Exploration（探索）** | 在 Feed 中随机插入 1-2 个非推荐类型视频 | 发现用户新兴趣，更新兴趣模型，提升长期留存 |

## 九、架构总结（面试收尾模板）

```
【上传流程】
用户 → API Gateway → 业务服务（保存 metadata，返回 Presigned URL）
     → 直传 S3 → 触发消息队列 → 转码服务（分块并行，多画质 HLS）
     → 更新 metadata DB（status=ready）+ KV（用户视频列表）

【Feed 播放流程】
用户 → API Gateway → 推荐系统（召回+排序，Top K）
     → Redis 缓存（热门 metadata，Hot Key 加 replica）
     → 返回 Feed（含 CDN Streaming URL）
     → 客户端预加载 HLS chunk → 自适应码率播放

【关键技术选型】
- 视频存储：S3（对象存储）+ CDN（GeoDNS 分发）
- 元数据：关系型 DB + video_id 哈希分片 + 读写分离
- 用户视频列表：KV 存储（反范式，消息队列保最终一致性）
- 缓存：Redis（热门 metadata，单飞锁防 Cache Stampede）
- 转码：消息队列触发，分块并行异步处理
- 推荐：召回（规则过滤）+ 排序（ML 模型，完播率为核心指标）

【数字驱动】5 亿 DAU → 需要分库分表 + Redis + CDN
           100 亿视频 → video_id 哈希分片避免热点
           99.99% 可用性 → 多 AZ 部署 + DB replica + CDN 冗余
```

## 十、术语速查

| 术语 | 中文 | 一句话解释 |
|------|------|-----------|
| Presigned URL | 预签名 URL | 由服务器签发的临时上传/下载链接，客户端直传 S3，绕过业务服务器 |
| HLS | HTTP 直播流 | 视频切成小 chunk 的流媒体协议，支持边下边播 |
| ABR | 自适应码率 | 根据实时网速动态切换视频分辨率 |
| Prefetching | 预取 | 提前下载用户可能观看的内容到 CDN 或客户端缓存 |
| Transcoding | 转码 | 将原始视频转成多种格式和画质（360p/720p/1080p + HLS） |
| Denormalization | 反范式 | 冗余存储数据（如 KV 存用户视频列表），用写入代价换查询效率 |
| Request Coalescing | 请求合并 / 单飞 | 多个并发请求只允许一个去数据库，其余等待同一结果 |
| Recall | 召回 | 推荐系统第一阶段：用简单规则从海量内容快速筛出候选集 |
| Ranking | 排序 | 推荐系统第二阶段：用复杂模型对候选集精细打分，取 Top K |
| Completion Rate | 完播率 | 用户看完视频的比例，推荐系统最核心的隐式信号 |
| Filter Bubble | 信息茧房 | 只推用户已知喜好，导致用户视野越来越窄 |
| Exploration vs Exploitation | 探索 vs 利用 | 推荐系统在挖掘已知兴趣和发现新兴趣之间的权衡 |

## 十一、面试答题习惯提醒

> AI 反馈：本题三个需主动带入的习惯：

1. **数字驱动设计**：每个技术决策要和需求数字挂钩。不要说"需要分库分表"，要说"**因为 5 亿 DAU + 100 亿视频，单表撑不住，所以按 video_id 哈希分片**"。

2. **收尾时主动覆盖非功能需求**：总结时要提 99.99% 可用性怎么保证（多 AZ 部署、DB replica、CDN 冗余），不要只说功能流程。

3. **Cache Stampede 解法要说全**：说了 Circuit Breaker 后，主动补充更轻量的 Request Coalescing（单飞锁）——一个请求去 DB，其余等结果，比熔断更直接。
