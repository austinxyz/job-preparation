# 系统设计复习：缓存与一致性

> 覆盖主题：缓存淘汰策略 → 缓存一致性 → Cache Stampede → CAP理论 → 朋友圈推拉模式 → Kafka扩展 → Load Shedding → 三者取舍框架

---

## 一、缓存基础

| # | 问题 | 核心答案 | 关键词 / 扩展点 |
|---|------|----------|----------------|
| 1 | 为什么不把所有数据放缓存？ | 缓存空间有限、价格远高于磁盘；生产系统数据量可达亿级，全放财务上不可行 | RAM vs Disk 成本比；缓存是热数据子集 |
| 2 | LRU 策略下，重要但冷门的数据会被踢出，怎么处理？ | ① 将重要数据和普通数据放**不同缓存实例**（最简洁）；② 外部程序定期访问维持热度（hack，会浪费资源） | 分类隔离优于复杂权值调优 |
| 3 | LRU vs LFU vs 加权打分，如何选择？ | LFU 按访问频率淘汰；加权打分可综合重要性和频率，但维护成本高；**最简方案：按类型隔离缓存** | LRU: Least Recently Used；LFU: Least Frequently Used |
| 4 | 缓存有哪些常见淘汰策略？ | LRU（最久未访问）、LFU（最低频率）、TTL（过期时间）、Random、分类隔离 | 面试直接给出 trade-off：LRU 简单但不感知重要性 |

## 二、缓存一致性

| # | 问题 | 核心答案 | 关键词 / 扩展点 |
|---|------|----------|----------------|
| 5 | 数据库更新后，如何保持缓存一致？ | **Cache-Aside（旁路缓存）**：先删缓存 → 更新DB → 再删缓存（Double Delete）；**Write-Through**：只写缓存，由缓存异步同步DB | Double Delete 是最常用方案 |
| 6 | "先删缓存再更新DB"有什么竞争条件？ | 线程A删缓存 → 线程B读缓存miss→从DB读旧值→写回缓存 → 线程A写新值入DB → **缓存仍是旧值** | 需要 Double Delete：更新DB后再删一次缓存 |
| 7 | Double Delete 第二次删除失败怎么办？ | **TTL 作为最终兜底**：即使第二次删除失败，缓存也会在 TTL 到期后自动失效；重要且频繁更新的数据设置**短 TTL** | 兜底 = TTL；短 TTL 的代价 = Cache Hit Rate 下降 |
| 8 | 什么是 Cache Stampede？如何解决？ | 大量请求同时发现缓存过期，同时打穿到数据库。解决：① **加缓存锁（Single Flight / Request Coalescing）**只让一个请求查DB；② **多读副本**分散并发；③ **提前异步刷新**（在TTL到期前后台预刷新） | Single Flight 是 Go 标准库 `singleflight` 的实现 |
| 9 | 缓存锁本身会成为瓶颈吗？ | 会：其他请求阻塞等待，线程资源耗尽风险。对策：① **分片多副本**降低单点压力；② **非阻塞返回**（返回旧值或等待提示，让客户端重试） | Non-blocking 策略 = 返回 stale data 或 retry hint |

## 三、一致性模型

| # | 问题 | 核心答案 | 关键词 / 扩展点 |
|---|------|----------|----------------|
| 10 | 强一致性 vs 最终一致性，如何选？ | **业务决定**：数据出错代价大（钱、库存）→ 强一致；数据晚点到没关系（点赞数、动态）→ 最终一致 | 转账 = 强一致；朋友圈好评数 = 最终一致 |
| 11 | 库存秒杀场景，最终一致性会超卖，怎么办？ | 用 **Redis 原子操作**保证强一致：`SETNX` 或 Lua 脚本，单线程执行，性能依然高 | Redis 单线程模型 + Lua 脚本保证原子性；SETNX = SET if Not eXists |
| 12 | CAP 定理是什么？怎么用于做设计决策？ | C（一致性）、A（可用性）、P（分区容错）三者只能满足两个。大系统必须有P，所以是 **CP vs AP** 的选择。**三个判断问题**：① 数据错了代价？② 不可用代价？③ 峰值多极端？ | CP 例子：ZooKeeper；AP 例子：Cassandra、DynamoDB |
| 13 | 朋友圈选 CP 还是 AP？为什么？ | 选 **AP**：朋友圈动态晚几秒看到可以接受，但系统不可用用户无法刷新体验更差 | 最终一致性 SLA：产品定时间上限 → 技术监控 metric → 超标 = incident |
| 14 | "最终"一致性里的"最终"有时间上限吗？ | 有。由**产品团队定义 SLA**（如：99% 消息 3 秒内送达），技术团队用 metric 监控，超标触发 incident 处理 | SLA（Service Level Agreement）闭环 = 产品定标准 + 技术实现 + 监控保障 |

## 四、朋友圈 Feed 系统（推拉模式）

| # | 问题 | 核心答案 | 关键词 / 扩展点 |
|---|------|----------|----------------|
| 15 | 朋友圈 Feed 推模式 vs 拉模式，区别？ | **推（Fan-out on Write）**：发动态时，将消息写入每个关注者的收件箱；**拉（Fan-out on Read）**：读 Feed 时，查询所有关注者的动态再合并 | 写扩散 vs 读扩散 |
| 16 | 什么情况下用推，什么情况下用拉？ | **普通用户用推**（粉丝少，写开销可控，读延迟低）；**名人用拉**（5000万粉丝，推写5000万次不可行） | 微信朋友圈：混合模式。粉丝数阈值（如 10万）作为切换条件 |
| 17 | 推模式下名人发动态（5000万粉丝）怎么处理？ | 用 **Message Bus（Kafka）异步处理**：发动态触发消息 → Kafka → Consumer 异步写收件箱，避免同步写5000万次阻塞 | 同步写 = 用户等待；异步写 = 延迟但不阻塞 |
| 18 | 拉模式的性能问题？ | 每次刷新需查询所有关注者的动态表，关注越多名人查询越多；可以用**缓存 + 预计算**优化（读时聚合 → 缓存聚合结果） | N+1 查询问题；需要缓存聚合层 |

## 五、Kafka 与流量峰值

| # | 问题 | 核心答案 | 关键词 / 扩展点 |
|---|------|----------|----------------|
| 19 | 为什么选 Kafka 而不是普通消息队列？ | Kafka 支持**水平扩展**（Partition 分片处理不同消息）+ **高可用**（Replication 多副本）+ **消费者组**（不同下游系统独立消费同一消息） | Partition = 并行度；Replication = 容错；Consumer Group = 扇出 |
| 20 | 跨年夜流量100倍，Kafka 积压怎么办？ | ① **Auto Scale**：监控队列长度，动态增加 Consumer；② **用户分级**：在线用户走高优先级队列，离线用户低优先级慢处理；③ **Load Shedding**：主动丢弃部分消息 | 扩 Partition 需要时间且不解决已有积压；分级是更快的响应手段 |
| 21 | 什么是 Load Shedding？什么时候用？ | 主动丢弃部分请求/消息，保证核心服务可用。**适用**：内容资讯类（AP系统，丢几条动态可接受）。**不适用**：金融交易类（CP系统，每笔交易必须处理） | Load Shedding = 可控降级；优先保核心链路 |

## 六、总结框架

### 性能 vs 一致性 vs 成本 的三问取舍框架

| 问题 | 答案方向 | 例子 |
|------|----------|------|
| **① 数据错了代价是什么？** | 代价大（钱/库存）→ 强一致 | 转账、秒杀库存 |
| **② 系统不可用代价是什么？** | 不可用比旧数据更糟 → 选 Availability | 朋友圈、社交 Feed |
| **③ 峰值流量多极端？** | 有明显峰值 → 提前设计 Load Shedding + 降级策略，不能靠硬撑 | 跨年夜、秒杀活动 |

> **面试金句**："系统设计里没有银弹，取舍的核心是业务特性。我会先问这三个问题，然后根据答案决定一致性模型和扩展策略。"

## 七、知识点速查

| 术语 | 中文 | 一句话解释 |
|------|------|-----------|
| LRU | 最近最少使用 | 踢出最久没被访问的数据 |
| LFU | 最低频率使用 | 踢出历史访问次数最少的数据 |
| TTL | 过期时间 | 缓存数据的生命周期，到期自动失效 |
| Cache Stampede | 缓存雪崩 | 大量缓存同时失效，请求涌入DB |
| Single Flight | 单飞 | 同一时刻只允许一个请求查DB，其余等待结果 |
| Cache-Aside | 旁路缓存 | 应用代码负责维护缓存，读miss则查DB写回缓存 |
| Write-Through | 写穿透 | 写时同步更新缓存和DB |
| Double Delete | 双删 | 更新DB前后各删一次缓存，减少不一致窗口 |
| CAP | 一致性/可用性/分区 | 三选二；大系统必须有P，所以是CP vs AP |
| Fan-out on Write | 推模式 | 写时扩散到所有收件箱 |
| Fan-out on Read | 拉模式 | 读时聚合所有关注者的动态 |
| Load Shedding | 负载降级 | 主动丢弃部分请求，保护核心服务 |
| SLA | 服务等级协议 | 产品层定义的可用性/延迟指标承诺 |
| SETNX | Redis 原子操作 | SET if Not eXists，常用于分布式锁和库存扣减 |
