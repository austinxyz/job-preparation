# 第二部分：数据平面技术

## 数据平面谱系

数据平面负责：设备暴露、资源分片、隔离机制、QoS 实现——不处理调度队列或配额。

**五种共享机制**（从强隔离到软约束）：

| 机制 | 特点 |
|------|------|
| 独占直通（Pass-through）| 最强隔离，利用率最低 |
| 硬分区（MIG）| 硬件级隔离，离散资源单位，重配置成本高 |
| 中介虚拟化（vGPU）| 软件级强隔离，运维复杂度高 |
| 时间切片（Time-slicing）| 超卖灵活，引入调度干扰和尾延迟风险 |
| MPS（Multi-Process Service）| HPC 并发优化，非多租户隔离语义 |

**CDI（Container Device Interface）**：标准化设备交付给容器，将机制选择与运行时集成解耦。

有效平台的三层组合：控制面治理 + 数据平面机制 + 可观测性/计量。

## MIG（Multi-Instance GPU）

MIG 解决的核心问题：在多租户环境中提供稳定的 P95/P99 尾延迟保障，消除"吵闹邻居"问题。

**两层架构**：
- GI（GPU Instance）：大分片，包含显存和 GPU 资源
- CI（Compute Instance）：GI 内部更细粒度的算力单元

**三种隔离机制**：
1. 显存隔离：防止 OOM 级联和跨租户显存争用
2. 缓存/片上资源隔离：降低共享资源引发的长尾延迟
3. 算力隔离：最小化 SM 竞争

**两大代价**：
1. **资源碎片化**：内部碎片（需要 9GB 但只有 10GB 剖面）+ 组合碎片（剩余剖面无法匹配新负载）+ 集群级倾斜（异构节点 MIG 几何形状导致持续队列延迟）
2. **重配置开销**：严重低估——"MIG 模式/几何形状变更需要目标 GPU 上零用户工作负载"，往往需要 cordon/drain 或节点重启

**推荐工程策略**："把 MIG 作为节点池属性固化"，用亲和性路由匹配工作负载。将每个节点池的 MIG 几何形状视为不可变，通过 label 和调度策略分配——而非动态重配置。

K8s 集成：三种策略（none / single / mixed），推荐 single 保持一致性。

**适合**：多租户推理（稳定负载形状）、共享研究集群（需要隔离）、MLOps 平台（强资源边界）
**不适合**：工作负载形状频繁变化、极度细粒度弹性优先的场景

## HAMi

**定位**：声明式、可调度、可治理的 GPU 共享数据平面，将 GPU 共享从"能跑就行"提升为可工程化的能力。

**四层分配链**：
1. **节点发现与上报（Device Plugin）**：识别 GPU，将物理卡映射为细粒度可共享资源单位，暴露自定义资源维度给调度器
2. **调度与放置（Scheduler 扩展）**：将请求路由到特定卡/份额，考虑拓扑和碎片化，将数据平面约束转化为调度可见约束
3. **容器级执行（vGPU 控制）**：在容器内兑现声明的份额，使用算力时间片 + 显存配额
4. **可观测性与计量（vGPU 监控）**：将共享状态指标导出为 Prometheus 数据

**资源表达创新**：支持声明 GPU 切片 / 算力分数 / 显存配额，而非二元"整卡"决策。

**关键区别**：HAMi 提供"比整卡分配更好的治理"（策略层），而非"MIG 级硬件分区"（硬隔离层）。

HAMi（数据平面）+ Volcano/Kueue（控制面）互补：HAMi 定义"如何划分和兑现 GPU 份额"，控制面强制执行"哪个工作负载获得哪个份额"。
