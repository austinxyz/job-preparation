# 第三部分：控制面治理

## 生态放置法

评估 GPU 方案的核心原则："GPU 方案争论的根源，不在'谁更强'，而在'谁解决的是同一层的问题'"。

**三层架构**（评估时先定位层级）：
- **数据平面**：资源呈现、隔离、观测（MIG、vGPU、时间切片）
- **控制面**：资源供给的治理与策略（Kueue、Volcano）
- **平台层**：能力组合与产品化体验（驱动交付、多集群、标准化运维）

四步放置法：
1. 固定目标工作负载与 SLO
2. 明确资源单位（整卡/分片/共享）
3. 定位执行点（驱动/运行时/调度器）
4. 写清责任边界（"我负责什么，不负责什么"）

典型组合：
- 离线训练优先 → MIG + Volcano（硬隔离 + Gang 调度）
- 在线推理优先 → HAMi + Kueue（声明共享 + 配额管理）
- 混合负载 → 分池管理

## Volcano

**定位**：批处理语义 + AI 训练作业调度的控制面调度引擎。解决原生 K8s 在 AI 训练和批处理的调度语义缺口。

**三层架构**：
- Job/Task 层：将分布式训练组件（worker、参数服务器）组合为可调度单元
- PodGroup 层：实现 Gang 调度——确保最小副本数后才执行，防止部分启动
- Queue/Quota/Priority 层：显式准入控制、公平份额分配、优先级访问

**核心调度能力**：
- **Gang 调度**：全有或全无执行，防止资源僵局
- **Backfill**：用小作业填充空隙，不阻塞高优先级作业
- **抢占**：可解释的优先级驱逐
- **DRF/公平份额**：多维资源公平性
- **拓扑亲和性**：为通信密集型工作负载共同优化放置

**Volcano 明确不处理**：GPU 虚拟化、隔离、资源切分——这些属于数据平面（MIG、HAMi）。两层正交可组合。

**有效部署的三个前提**：
- 可验证的队列语义（避免"待处理黑洞"）
- 可观测的抢占决策（驱逐审计轨迹）
- 闭环利用率追踪（确认实际 GPU 效率提升，而非只是队列排序）

## Kueue

**定位**：资源治理与准入控制——从"工作负载能否运行"到"如何公平治理资源分配"。两阶段调度：**先准入，再调度**。

**四个核心组件**：
- **LocalQueue**：团队/命名空间的作业提交入口
- **ClusterQueue**：全局资源池，定义配额、借用规则和公平策略
- **Workload**：从 Job 映射的可管理资源请求单元
- **ResourceFlavor**：显式建模资源差异（H100 vs A100、MIG 剖面），支持差异化配额管理

**七项核心治理能力**：配额管理、准入控制、借用机制、优先级抢占、Cohort 队列共享、资源类型差异化、跨租户公平分配。

**关键设计**："在资源真正分配到节点和设备之前，先完成资源的准入与承诺"——让待处理原因透明化，减少后期失败。

**与 Volcano 的关系**（正交互补）：
- Volcano：如何排座位（Gang 调度、批处理语义）
- Kueue：谁能进场、能占多少座位（准入、配额治理）

**Kueue 的边界**：不提供隔离（数据平面职责）、不替代调度器、无法完全解决碎片化（需要数据平面配合）。

## 六种组合参考架构

| 方案 | 数据平面 | 控制面 | 适用场景 |
|------|----------|--------|----------|
| A | 整卡 | 单队列 | 最小可行闭环 |
| B | 整卡 | Kueue 多队列 | 引入公平治理 |
| C | MIG | Kueue | 强隔离 + 细粒度 |
| D | HAMi | Kueue | 高利用率 + 声明共享 |
| E | 整卡/HAMi | Kueue + Volcano | 训练语义增强 |
| F | 双池（在线/离线） | 分域管理 | 在线离线隔离 |

**四条组合原则**：
1. 优先建立最小可行闭环，而非追求最大功能集
2. 资源单位越细，验收标准越严格
3. 重配置（尤其 MIG）必须走正式变更流程
4. 清晰划分数据平面、控制面、平台层的责任

**三层验收指标**：
- 调度层：队列等待时间、成功率、抢占事件
- 性能层：吞吐/延迟百分位（P50/P95/P99）、抖动边界
- 运营层：使用归因、承诺兑现率、违约频率

**推荐演进路径**：Phase 1（建立秩序：A→B）→ Phase 2（引入粒度：B→C/D）→ Phase 3（平台成熟：→E/F）
