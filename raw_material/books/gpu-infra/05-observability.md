# 第五部分：可观测与验收

## 观测与计量

**核心命题**："只有能被解释、被治理、能结算的资源，才构成一个可运营的 GPU 平台。"

**平台运营的十个关键问题**：

资产与资源视角：
- 谁占用了哪块 GPU（UUID / MIG slice / vGPU）？
- 利用率水平如何（显存 vs 算力）？

组织与治理视角：
- 利用率为何低（碎片化 / 排队 / 负载限制）？
- 调度规则和配额是否被遵守？

性能与体验视角：
- 性能波动源于哪里（争用 / 邻居噪声 / 负载方差）？
- SLA/SLO 目标是否达成？影响了谁？

稳定性视角：
- 故障率和恢复时间（OOM / 驱动 / MIG / 启动）？
- 资源是否被超用或规避？

财务与容量视角：
- 如何按租户/团队/项目分摊成本？
- 未来时段有哪些容量缺口？

**三级实施路径**：
- **Level 0**（1-2 天）：基础 nvidia-smi，回答 4 个核心问题
- **Level 1**（1-2 周）：DCGM Exporter + Prometheus + Grafana，回答 6 个问题
- **Level 2**（1-2 月）：完整可审计性，分配/使用对账，回答全部 10 个问题

**六个常被混淆的定义**：
- 已分配 vs 已使用（治理视角 vs 性能视角）
- 利用率 vs 饱和度（活跃度 vs 需求压力）
- 碎片化：不是"剩余显存少"，而是"剩余资源无法匹配典型请求形状"
- 队列等待时间（优先级 KPI，优先于平均利用率）
- 干扰系数（共享场景下的性能降级）
- 标准化 GPU-小时（统一 MIG / vGPU / 整卡计费）

**最小可行平台验收标准**（7/10 条）：GPU 到工作负载的可追溯性、团队级成本报告、队列延迟百分位、按根因分类的故障率。

## 验收与容量

**基准测试三类**（不能混用）：
1. 峰值性能：硬件健康检查和回归基线
2. 可共享性：争用下的多租户稳定性
3. 可交付性：含排队和故障的真实 SLO 达成

**四象限验收框架**：
- 性能：吞吐、P50/P99 延迟、冷启动时间
- 稳定性：P99 抖动、故障率、MTTR
- 效率：碎片化率、利用率、能耗成本
- 治理：队列等待时间、拒绝率、配额遵从度

**实验设计**：三组并行测试路径——隔离基线 / 非托管共享（风险演示）/ 托管共享（控制证明）/ MIG 隔离（权衡分析）。

**容量规划三问**：
1. 我们需要服务什么需求？
2. 每块 GPU 满足 SLO 的吞吐是多少？
3. 需要多少安全边距（抖动、故障、治理）？

最终输出：带可解释 SLO 保障的资源承诺模型。

## 排障

**核心原则**："排障的本质是用系统化方法缩短定位路径，而不是靠经验'拍脑袋'。"

**四问快速分诊**：
1. 现象类型（Pod Pending / GPU 不可见 / OOM / 性能降级 / 资源不一致）
2. 爆炸半径（单 Pod / 节点 / 队列 / 集群）
3. 最近变更（驱动/CUDA 更新、MIG 重配、调度器修改）
4. 可重现性（稳定 vs 间歇）

**四阶段收敛流程**：
1. 层级判断：控制面（调度规则）vs 数据平面（资源可用性）
2. 问题分类：配置/运维 vs 工作负载行为
3. 证据收集：用最小命令集建立证据链
4. 最小修复：最小化变更 + 回归验证

**五种主要故障模式**：
- Pod Pending：查 `kubectl describe` Events，验证节点 GPU 容量，确认调度器健康
- GPU 不可见：验证宿主机 GPU 正常 → 容器内 nvidia-smi → 运行时 hook 配置（nvidia-container-toolkit）
- 显存 OOM：区分工作负载驱动（批次大小、KV cache）vs 隔离/配额问题
- 性能异常：先排除 CPU/网络瓶颈，再对比隔离基线 vs 共享场景
- MIG 配置：验证节点与 K8s 之间实例状态一致性，避免峰期重配置

**知识积累**：每次事故转化为组织资产——触发条件、根因分类、预防机制、回归测试用例。
