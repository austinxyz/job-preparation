# 第四部分：工作负载实践

## vLLM：推理吞吐 vs 尾延迟

**核心命题**："推理平台的真正挑战，不在于极限吞吐，而在于如何用资源契约守住尾延迟的底线。"

**根本张力**：吞吐优化（tokens/s、QPS）与尾延迟（P95/P99）之间的非线性权衡——两个物理瓶颈：
1. 调度队列等待时间（请求执行顺序与批处理）
2. KV cache 占用与碎片化（显存压力限制并发序列数）

**vLLM 资源模型的隐藏代价**：
- 更高并发 → 更大 KV cache 占用
- 更长 context/max_model_len → 更高的每序列显存上限
- 更长输出 → cache 驻留时间延长
- Continuous batching：提升吞吐但增加排队方差；短请求因等待填充大批次而受损

**三层资源配额框架**：
1. 硬资源：GPU 分配 + 显存余量（不是"浪费"，是防止非线性崩溃的保险）
2. 软配额：最大并发序列数 + 上下文/输出长度上限（通过准入控制）
3. 体验配额：Goodput（SLO 约束的吞吐）而非原始 tokens/s

**五类高风险配置**：
- 显存利用率过高 + 高并发 → 触发碎片级联
- 混合工作负载无队列隔离 → 短请求被长请求阻塞
- 无界上下文长度 → 罕见超长序列使整个系统不稳定
- 多租户共享隔离不足 → 不可见的邻居干扰
- 吞吐最大化与 SLO 脱钩 → 不稳定性转移给用户

**验收指标四维度**（P50/P95/P99 分布）：
- **TTFT**（首 token 时间）：反映队列健康
- **TPOT**（每输出 token 时间）：生成阶段稳定性
- **Throughput + Goodput**：tokens/s 单独会误导，SLO 合规吞吐才是真实容量
- **可靠性**：拒绝/OOM/重试率是崩溃前的预警信号

核心洞察："吞吐优化的收益是平均意义上的，尾延迟是最坏情况的。"没有 P99 可见性的验收，问题会迁移到生产。

## PyTorch：调度语义与抢占影响

**训练 vs 推理的根本差异**：推理优先响应时间；训练强调持续吞吐和通信拓扑稳定性。

**训练资源四大特性**：
1. **连续性**：训练跨小时/天，任何中断都有回滚和缓存重建成本
2. **同步通信**：分布式训练（DDP/FSDP）高度依赖集合操作，对拓扑和网络方差敏感
3. **刚性资源单位**：需要特定数量和特定拓扑的 GPU，部分分配浪费资源
4. **耦合约束**：显存、I/O、算力相互作用，优化涉及多维权衡

**工作负载三分类**：
- **A 类**（单机微调）：快速迭代，低开销，标准队列
- **B 类**（多节点分布式）：需要 min-available 保障、Gang 调度、拓扑感知
- **C 类**（弹性训练）：容忍伸缩，适合机会性资源

**五种必需的调度语义**：
1. Min-Available：防止部分启动浪费 GPU 周期
2. Gang 调度：确保同时分配和拓扑一致性
3. 拓扑感知：优先同节点 > 同机架 > 跨域
4. 可抢占性分层：区分可中断 vs 保障型工作负载
5. Suspend/Resume：比终止重启更温和的替代方案

**抢占的真实代价**（不只是资源回收）：
- 重启开销：容器启动、权重下载、通信组初始化
- 回滚损失：checkpoint 频率决定恢复代价
- 系统抖动：不稳定的抢占策略触发级联中断，降低实际吞吐（即使 GPU 利用率显示很高）

**推荐治理架构**：
- **Kueue**：准入控制、配额、公平性、全局承诺
- **Volcano**：Gang 调度、拓扑感知、策略执行
- **数据平面**：定义资源单位和隔离边界

**核心洞察**："你优化的不是 GPU 可用性，而是在资源争用下的有效步数产出（valid step production）。"成功从"启动速度"重新定义为"持续、抗中断的训练进展"。

**关键风险点**：
- 多节点作业必须强制 min-available + Gang，否则资源僵局
- 拓扑不可控会产生不可预测的性能——需要明确 SLO 说明
- 无 checkpoint 纪律的抢占使中断具有破坏性
- 恢复触发在无冷却机制时会造成队列振荡
- 推理/训练共享资源池产生根本性压力冲突——需要显式隔离
